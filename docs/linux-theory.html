🐧 Linux Theory

The concepts behind the commands — how Linux actually works.

    1What is Linux?

    Linux is an open-source kernel — the core program that talks to the hardware and manages CPU, memory, devices, and processes. It was started by Linus Torvalds in 1991.

    What people casually call "Linux" is really a distribution (distro): the Linux kernel bundled with GNU tools, a package manager, and userland software. Examples: Ubuntu, Debian, Fedora, CentOS/RHEL, Arch, Alpine.

    • Kernel — schedules processes, manages memory, drives hardware, enforces permissions.
    • Userland — the shell, coreutils (ls, cp…), libraries, and apps that run on top.
    • Distro — a curated, installable package of kernel + userland + defaults.
    💡 A guiding Unix philosophy: everything is a file, programs do one thing well, and text is the universal interface — which is why the shell and pipes are so powerful.

    2The Filesystem Hierarchy

    Linux has a single tree starting at the root / — there are no drive letters like C:. Storage devices are mounted into this tree. The layout follows the Filesystem Hierarchy Standard (FHS):

    PathPurpose
    /Root of everything
    /bin, /usr/binEssential user command binaries
    /sbinSystem/admin binaries
    /etcSystem-wide configuration files (text)
    /homeUsers' personal directories
    /rootThe root user's home
    /varVariable data: logs, spools, caches
    /tmpTemporary files (often cleared on reboot)
    /devDevice files (disks, terminals)
    /proc, /sysVirtual filesystems exposing kernel/process state
    /mnt, /mediaMount points for external filesystems
    /optOptional/third-party software
    📁 "Everything is a file": disks, terminals, even running processes appear as files under /dev, /proc, and /sys. You can read hardware and kernel state with the same tools you use for text files.

    3Users, Groups & Permissions

    Linux is multi-user. Every file has an owner, a group, and a set of permissions for three classes: user (u), group (g), and others (o).

    Each class can have three permissions: r (read), w (write), x (execute). In ls -l you see them as a 10-character string like -rwxr-xr--:

    • Char 1 = type (- file, d directory, l symlink).
    • Chars 2–4 = owner, 5–7 = group, 8–10 = others.

    Numeric (octal) mode

    Each permission is a bit: r=4, w=2, x=1. Add them per class:

    • chmod 755 file → rwx for owner, r-x for group & others.
    • chmod 644 file → rw- for owner, r-- for the rest.

    Special bits & root

    • setuid (4xxx) — run a program with the owner's privileges (e.g. passwd).
    • setgid (2xxx) — run with the group's privileges; on a dir, new files inherit the group.
    • sticky bit (1xxx) — on /tmp, only the file's owner can delete it.
    • root (UID 0) — the superuser, bypasses permission checks. Use sudo for one-off elevation.

    4Processes & Signals

    A process is a running program with its own PID (process ID), memory, and file descriptors. Processes form a tree — each has a parent (PPID). The first process at boot (PID 1) is init/systemd, the ancestor of everything.

    • Foreground vs background — end a command with & to run it in the background; jobs, fg, bg manage them.
    • States — running, sleeping, stopped, or zombie (finished but not yet reaped by its parent).

    Signals

    Processes are controlled by signals — asynchronous notifications:

    SignalNumberMeaning
    SIGTERM15Politely ask to terminate (default of kill)
    SIGKILL9Force kill, cannot be caught (kill -9)
    SIGINT2Interrupt from keyboard (Ctrl+C)
    SIGHUP1Hangup; often used to reload config
    SIGSTOP / SIGCONT19 / 18Pause / resume a process

    5The Shell, Streams & Pipes

    The shell (usually bash or zsh) reads your commands and runs programs. Every process gets three standard streams:

    • stdin (0) — input.
    • stdout (1) — normal output.
    • stderr (2) — error output.

    Redirection & pipes

    • cmd > file — send stdout to a file (>> appends).
    • cmd 2> err.log — redirect stderr; &> all.log redirects both.
    • cmd1 | cmd2 — pipe: feed cmd1's stdout into cmd2's stdin.

    This composability — small tools chained with pipes — is the heart of working productively on Linux.

    Environment & exit codes

    • Environment variables (PATH, HOME…) configure how programs behave; export VAR=value passes them to child processes.
    • Exit code — every command returns 0 for success, non-zero for failure, readable via $?. Scripts branch on this.

    6The Boot Process

    From power-on to login prompt, Linux goes through well-defined stages:

    • 1. Firmware (BIOS/UEFI) — initializes hardware, finds a bootable device.
    • 2. Bootloader (GRUB) — loads the kernel and an initial ramdisk (initramfs) into memory.
    • 3. Kernel — initializes hardware/drivers, mounts the root filesystem, starts PID 1.
    • 4. init / systemd — the first user-space process; starts services and brings the system to its target state.
    • 5. Login — a getty or display manager presents a login prompt.

    7systemd & Services

    Most modern distros use systemd as PID 1 to manage services (daemons), mounts, timers, and logging. A managed thing is a unit; a background service is a .service unit.

    • systemctl start|stop|restart nginx — control a service now.
    • systemctl enable|disable nginx — start (or not) automatically at boot.
    • systemctl status nginx — current state and recent logs.
    • journalctl -u nginx -f — follow that service's logs (systemd's journal).
    ⚙️ enable ≠ start. enable sets it to run at boot; start runs it right now. You often want both.

    8Package Management

    Software is installed from repositories via a package manager, which resolves dependencies for you. The tool depends on the distro family:

    FamilyManagerInstall example
    Debian/Ubuntuapt (.deb)apt install nginx
    RHEL/Fedoradnf/yum (.rpm)dnf install nginx
    Archpacmanpacman -S nginx
    Alpineapkapk add nginx

    Cross-distro options like Snap, Flatpak, and language managers (pip, npm) coexist with the system package manager.

    9Networking Basics

    Linux is the backbone of the internet, so networking is central:

    • Interfaces — lo (loopback, 127.0.0.1), eth0/ens… (wired). Inspect with ip a.
    • IP & CIDR — an address plus a prefix like /24 defines the network vs host bits.
    • Ports — a service listens on a port (HTTP 80, HTTPS 443, SSH 22). See them with ss -tulpn.
    • DNS — resolves names to IPs; test with dig or nslookup.
    • Firewall — the kernel's netfilter, managed via iptables/nftables or friendly ufw.

    10Kernel, /proc & /sys

    The kernel runs in a protected kernel space; your programs run in user space and ask the kernel for services via system calls (open, read, write, fork, exec…).

    • Drivers & modules — hardware support; loadable modules (lsmod, modprobe) extend the kernel at runtime.
    • /proc — a virtual filesystem exposing per-process and kernel info (e.g. /proc/cpuinfo, /proc/<pid>/).
    • /sys — exposes devices and kernel tunables; sysctl reads/sets kernel parameters.
    🧠 Containers (Docker, Kubernetes) are just Linux processes isolated by kernel features — namespaces (what a process can see) and cgroups (how much it can use). Understanding these concepts is what makes the DevOps tooling click.
